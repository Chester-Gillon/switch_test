This gives some example results from running tests, using different PCs and operating systems.

For all tests:
a. The injection switch was a D-Link DES-3526. The DES-3526_injection_switch_config.txt describes the configuration.
b. The switch under test was a tp-Link TL-SF1024, which is an unmanged 24 port 10/100M switch.

Unless stated otherwise all ports 1-24 were connected between the injection switch and switch under test.

A D-Link DGS-2000-52 could be an alternative injection switch to increase the number of ports under test to 48.
From looking at the D-Link DGS-2000 Ver.A 1 Managed Access Switch CLU Reference Guide v1.0 the DGS-2000-52 supports the same
CLI commands as were used to configure the DES-3526.


== Laptop running linux achieved rate on all ports == 

For this a Intel(R) Core(TM) i5-7200U based LENOVO ideapad 510S laptop was used, with a "Realtek PCIe GBE Family Controller"
which enumerates as enp1s0.

Running Ubuntu 18.04.6 LTS with a 4.15.0-161-generic Kernel.

== Error due to cable unplugged when switch under test power on ==

Left the cable removed from port 8, but power cycled the switch under test which causes it to reset its MAC address table,
and have to re-learn.

The test results show:
a. For port 8 the test has has been correctly reported as missing all frames for all source and destination ports.
b. Of the other ports *some* frames are reported as missed, and there is also a large number of "Rx Flooded" frames reported.

   What is happening is that because the switch under test has never seen any frames with the *source* MAC address for port 8,
   all the frames which have a *destination* MAC address for port 8 get flooded to all other ports than used for the source
   of the test frames. As a result of flooded frames the bandwidth of the 1G link from the injection switch back to the 
   controlling PC is exceeded and frames get lost.

Therefore, the results are valid and show how to a side effect of a failure which causes a large number of flooded frames
due to a port which has either never worked, or has failed after the switch under test has flushed its learned MAC addresses.
